Android vs iOS vs Webview

🔍 하이브리드 앱에서 적용되는 px 개념 파해치기



1️⃣ 화면 픽셀 기초 개념

주요 개념 KeyWord 정리

물리적 픽셀 (Physical Pixels):

디스플레이 화면에서 색상을 표현할 수 있는 가장 작은 개별 하드웨어 구성 요소(점)입니다. 이는 화면 매트릭스를 구성하는 실제적이고 유형적인 픽셀입니다.(1)
최종 렌더링 대상을 나타내며, 물리적 픽셀의 수는 화면이 표시할 수 있는 최대 세부 정보를 결정합니다.

논리적 픽셀 (Logical Pixels, Device-Independent Pixels):

화면의 물리적 픽셀 밀도와 관계없이 개발자가 UI 요소 크기 및 레이아웃을 정의하는 데 사용하는 추상 단위입니다.(1)
이 추상화를 통해 UI는 다양한 장치에서 일관된 크기로 표시됩니다.

플랫폼별 표현:
  • Android: 밀도 독립형 픽셀 (dp 또는 dip): 가상 픽셀 단위입니다. 1 dp는 160 DPI 화면에서 약 1 물리적 픽셀에 해당합니다.(6) Android 시스템은 화면 밀도에 따라 dp 단위를 실제 픽셀로 변환하는 작업을 처리합니다.(7)
  • iOS: 포인트 (Points): 화면 좌표 및 크기를 위한 추상 단위입니다. 시스템은 배율을 사용하여 포인트를 물리적 픽셀에 매핑합니다.(8)
  • Web: CSS 픽셀 (px): CSS 인치의 1/96로 정의됩니다.(9) 장치의 물리적 픽셀에 대한 배율 조정을 통해 여러 장치에서 시각적으로 거의 동일한 크기로 보이도록 의도된 표준 단위입니다.(3)
해상도 (Resolution):
  • 물리적 해상도: 화면의 총 물리적 픽셀 수로, 일반적으로 너비 × 높이(예: 1920x1080 픽셀)로 표현됩니다.(1)
  • 논리적 해상도: 논리적 단위로 표현된 화면 크기(예: iPhone X의 경우 375x812 포인트, Android 휴대폰의 경우 360x640 dp)입니다.(5) 개발자가 일반적으로 설계하는 해상도입니다.
PPI/DPI (Pixels Per Inch / Dots Per Inch):

물리적 픽셀 밀도의 측정 단위로, 화면의 1인치 선형 공간에 얼마나 많은 물리적 픽셀이 채워져 있는지를 나타냅니다.(1)
PPI/DPI가 높을수록 이미지가 더 선명하고 세밀해집니다. 기술적으로 “도트”는 프린터 출력을 의미할 수 있지만, 화면의 맥락에서 DPI와 PPI는 종종 물리적 픽셀 밀도를 의미하는 데 상호 교환적으로 사용됩니다.(3)
디스플레이의 이러한 물리적 특성은 OS 및 WebView에서 사용되는 배율을 결정하는 주요 입력값입니다.


2️⃣ 배율(Scale Factors)

각 플랫폼의 배율 속성이 동등해질 수 있는 이유(!?)

논리적 픽셀(dp, point, CSS 픽셀)의 크기와 물리적 픽셀 사이의 비율을 정의하는 숫자 값입니다. 시스템에 하나의 논리적 픽셀을 렌더링하는 데 사용해야 하는 물리적 픽셀 수를 알려줍니다.
픽셀 밀도가 다른 화면에서 UI 요소의 인지되는 물리적 크기를 유지하기 위함입니다.(4) 예를 들어, 배율 2.0은 하나의 논리적 픽셀이 2x2 물리적 픽셀 그리드(총 4개의 물리적 픽셀)를 사용하여 렌더링됨을 의미합니다.
세 가지 주요 플랫폼(Android, iOS, Web)은 개발자를 다양한 물리적 픽셀 밀도의 복잡성으로부터 보호하기 위해 독립적으로 논리적 픽셀 개념을 발전시켰습니다. 초기 모바일 장치는 상대적으로 균일하고 저밀도 화면을 가졌지만, 화면 기술이 발전함에 따라 픽셀 밀도가 급격히 증가했습니다.(1)

만약 UI가 여전히 물리적 픽셀로 설계되었다면, 요소들은 고밀도(High-DPI) 화면에서 축소되어 사용성 문제를 야기했을 것입니다.(3)
이에 대응하여 Android는 dp(6), iOS는 points(8), CSS는 px 단위(10)를 도입했으며, 이들 모두 디자인을 위한 안정적인 기준점 역할을 합니다.
이러한 공통된 접근 방식은 UI 개발의 근본적인 원칙, 즉 하드웨어 다양성 속에서 일관된 사용자 경험을 달성하는 것을 강조합니다. 배율은 이 추상화가 작동하도록 하는 수학적 다리 역할을 합니다.
이러한 공유된 전략은 각 플랫폼의 배율 속성(density, scale, devicePixelRatio)이 개념적으로나 수치적으로 동등해질 수 있는 근본적인 이유입니다.

3️⃣ Android의 접근 방식

Android가 DisplayMetrics.density 사용하여 디스플레이 배율을 관리하는 방법

DisplayMetrics:

디스플레이의 크기, 밀도, 글꼴 배율과 같은 일반적인 정보를 설명하는 구조체입니다.(15) 일반적으로 context.getResources().getDisplayMetrics()를 통해 얻습니다.

densityDpi (DisplayMetrics에서):

인치당 도트 수(DPI)로 표현되는 물리적 화면 밀도입니다.(15) 이 값은 화면의 실제 하드웨어 특성을 나타냅니다.
예시: 일반적인 값은 120(ldpi), 160(mdpi), 240(hdpi), 320(xhdpi), 480(xxhdpi), 640(xxxhdpi)과 같은 밀도 버킷에 해당합니다.(16)

DisplayMetrics.density:

디스플레이의 논리적 밀도입니다. 이는 밀도 독립형 픽셀(dp) 단위에 대한 배율입니다.(15)
계산: densityDpi에서 파생됩니다. 기준 밀도는 160 DPI (mdpi)이며, 여기서 density는 1.0입니다. 다른 밀도의 경우, density=densityDpi​/160 입니다.(6)
예시: hdpi 화면(240 DPI)의 경우, density=240​/160=1.5 입니다.
예시: xhdpi 화면(320 DPI)의 경우, density=320/160​=2.0 입니다.
시스템이 dp 단위를 실제 화면 픽셀 크기로 변환하여 UI 요소가 다양한 화면 밀도에서 일관된 크기로 보이도록 하는 데 사용됩니다.
X dp로 정의된 UI 요소는 화면에서 X × density 물리적 픽셀을 차지합니다.

dp, density, 물리적 픽셀 간의 관계:

dp = (width in pixels×160) / screen density (dpi)​ 공식(6)을 재정렬하면 pixels=(dp×screen density (dpi)) / 160​가 됩니다. density=screen density (dpi) / 160​이므로, 이는 pixels=dp×density로 단순화됩니다. 이는 X dp로 정의된 UI 요소가 화면에서 X × density 물리적 픽셀을 차지함을 의미합니다.

Android 밀도 버킷 (Density Buckets):

분류: ldpi, mdpi, hdpi, xhdpi, xxhdpi, xxxhdpi (6)
개발자가 다양한 밀도 범위에 최적화된 비트맵 애셋을 제공하도록 돕습니다. 시스템은 장치의 densityDpi를 기반으로 가장 적절한 애셋을 선택합니다.(7)

버킷별 density 값:
  • mdpi (160dpi): 1.0
  • hdpi (240dpi): 1.5
  • xhdpi (320dpi): 2.0
  • xxhdpi (480dpi): 3.0
  • xxxhdpi (640dpi): 4.0 (6)

정보 흐름 (OS에서 ➡️ DisplayMetrics로)⭐️⭐️⭐️:

Android OS는 디스플레이 하드웨어에서 해상도 및 물리적 크기를 포함한 물리적 특성을 조회합니다. 이를 통해 densityDpi를 계산합니다. 이 densityDpi와 파생된 density (및 widthPixels, heightPixels와 같은 기타 메트릭)가 DisplayMetrics 객체에 채워집니다. 애플리케이션과 Android 프레임워크는 DisplayMetrics를 통해 이러한 값에 접근할 수 있습니다.
밀도 버킷(mdpi, hdpi 등)은 리소스 관리를 위한 편리한 범주를 제공하지만, DisplayMetrics.density 자체는 화면의 실제 densityDpi에서 직접 파생된 부동 소수점 값이며, 버킷의 명목상 DPI에만 국한되지 않습니다.(6)
예를 들어, 장치가 200dpi의 densityDpi를 가질 수 있으며, 이 경우 DisplayMetrics.density는 200/160=1.25가 됩니다. 이는 버킷을 기준으로 1.0이나 1.5로 고정되지 않습니다. 31에서는 Nexus 7이 1.33 (tvdpi, 213dpi/160dpi)의 픽셀 비율을 가진다고 언급하며, 이는 정수가 아닌 density 값이 일반적으로 사용됨을 보여줍니다.
결과적으로 DisplayMetrics.density는 디스플레이 하드웨어의 실제 기능을 반영하는 세분화된 배율을 제공하여, 버킷의 불연속적인 단계에 엄격하게 제한되는 경우보다 더 정확한 UI 배율 조정을 가능하게 합니다. 이러한 정밀도는 WebView가 window.devicePixelRatio를 정확하게 결정하는 데 중요합니다.

4️⃣ iOS의 접근 방식

iOS가 포인트와 배율을 사용하여 디스플레이 배율을 관리하는 방법

UIScreen:

하드웨어 기반 디스플레이와 관련된 속성을 정의하는 객체입니다.(17) 주 디스플레이의 경우 일반적으로 UIScreen.main을 사용합니다.

포인트(Point) vs 픽셀(Pixels):

iOS는 화면 크기 및 레이아웃 좌표를 물리적 픽셀이 아닌 포인트(Point)로 참조합니다.(8) 포인트는 사용자 공간의 추상 단위입니다. 표준 해상도 디스플레이에서는 1 포인트가 1 물리적 픽셀에 매핑됩니다. 고해상도(Retina) 디스플레이에서는 1 포인트가 여러 물리적 픽셀에 매핑됩니다.(8)

UIScreen.main.scale:

해당 화면의 물리적 픽셀 대 포인트 비율을 나타냅니다.(17) 값: 일반적으로 1.0, 2.0 또는 3.0입니다.(18)

- @1x (scale 1.0): 1 포인트 = 1x1 물리적 픽셀.
- @2x (scale 2.0): 1 포인트 = 2x2 물리적 픽셀 (총 4개).
- @3x (scale 3.0): 1 포인트 = 3x3 물리적 픽셀 (총 9개).

개발자가 해상도 독립적인 좌표계(포인트)에서 작업할 수 있도록 합니다. 시스템은 배율을 사용하여 적절한 수의 물리적 픽셀을 활용하여 콘텐츠를 선명하게 렌더링합니다.

UIView.contentScaleFactor:

뷰는 자신이 속한 화면에서 배율을 상속받습니다. 이 속성은 뷰 콘텐츠가 논리적 좌표(포인트)에서 장치 좌표(픽셀)로 매핑되는 방식을 결정합니다.(23)

(UIScreen에서) nativeBounds 및 nativeScale:
- nativeBounds: 물리적 화면의 경계 사각형으로, 픽셀 단위로 측정됩니다.
- nativeScale: 물리적 화면의 기본 배율입니다.

bounds (포인트 단위) × scale = nativeBounds (픽셀 단위, 근사치이며 최신 장치의 주 디스플레이에서는 scale이 일반적으로 nativeScale과 동일함).
(11)은 UIScreen.scale이 UIKit 배율이며 때때로 기본 배율과 다를 수 있어 OS에서 중간 배율 조정 단계가 발생할 수 있다고 설명합니다. 그러나 대부분의 앱 개발에서 UIScreen.main.scale이 관련 요소입니다. (32)는 Metal 앱의 경우 리샘플링을 피하기 위해 nativeScale에서 렌더링할 것을 강조합니다.(17)

정보 흐름 (OS에서 UIScreen으로)⭐️⭐️⭐️:

iOS 운영체제는 연결된 디스플레이 하드웨어를 식별합니다. 하드웨어의 픽셀 밀도(예: Retina 대 비 Retina)를 기반으로 디스플레이의 물리적 픽셀 해상도(nativeBounds)와 고유한 배율(nativeScale)을 결정합니다. 이 정보는 UIScreen 속성을 통해 노출되며, UIScreen.main.scale은 포인트 단위 UI 렌더링을 위한 기본 배율을 제공합니다.

iOS의 scale 값은 일반적으로 1.0, 2.0, 3.0과 같은 깔끔한 정수 값을 가지며, 이는 Android의 잠재적으로 더 다양한 부동 소수점 density 값과 대조됩니다.
Apple은 하드웨어 생태계(iPhone, iPad)를 제어하므로(22), 이러한 정수 배율에 깔끔하게 매핑되는 특정 픽셀 밀도를 가진 디스플레이를 설계할 수 있습니다(예: “Retina” 디스플레이는 이전 @1x 디스플레이의 포인트 밀도의 @2x 또는 @3x가 되도록 설계됨). 이는 개발자의 애셋 관리를 단순화합니다(예: @1x, @2x, @3x 이미지 애셋 제공).(22)

반면 Android는 다양한 제조업체의 방대한 하드웨어에서 실행되므로 더 넓은 범위의 DPI가 발생하며, 이로 인해 더 유연한 부동 소수점 density 및 “밀도 버킷”이 필요합니다.(6) 결과적으로 두 시스템 모두 해상도 독립성을 달성하지만, iOS의 접근 방식은 통합된 하드웨어/소프트웨어 모델로 인해 더 규범적이며, 이는 더 단순하고 불연속적인 배율로 이어집니다. 이는 WKWebView가 이 배율을 수신하고 해석하는 방식에 직접적인 영향을 미칩니다.

5️⃣ Webview의 통합 요소

window.devicePixelRatio이 섹션에서는 장치 픽셀 밀도를 반영하는 웹 표준 속성에 대한 설명

window.devicePixelRatio:

현재 디스플레이 장치의 물리적 픽셀 해상도 대 CSS 픽셀 해상도의 비율입니다. 또는 CSS 픽셀 하나의 크기 대 물리적 픽셀 하나의 크기의 비율로 해석될 수도 있습니다. 브라우저에 단일 CSS 픽셀을 그리는 데 사용해야 하는 실제 화면 픽셀 수를 알려줍니다.(1)

값 1.0은 고전적인 96 DPI 디스플레이를 나타내며 (디스플레이가 실제로 96 DPI이고 OS 수준 배율이 적용되지 않은 경우 1 CSS 픽셀이 이상적으로 1 물리적 픽셀에 매핑됨), 값 2.0은 HiDPI/Retina 디스플레이에 예상됩니다. 최신 모바일 장치는 종종 2보다 큰 값을 산출합니다.(1)

반응형 웹 디자인을 가능하게 하는 방법:

웹 개발자가 장치의 픽셀 밀도를 조회하고 적절한 크기의 애셋(예: 고 DPR 화면용 고해상도 이미지)을 제공하거나 렌더링(예: HTML 캔버스용)을 조정할 수 있도록 합니다.(1) JavaScript(window.devicePixelRatio) 또는 CSS(resolution 미디어 쿼리 또는 비표준 -webkit-device-pixel-ratio 사용)에서 사용할 수 있습니다.(4)

<meta name="viewport" content="width=device-width, initial-scale=1">의 결정적 역할:

모바일 브라우저는 초기에 “가상 뷰포트”(실제 화면보다 넓음, 예: 980px)를 사용한 다음 페이지를 축소하여 보여줍니다. 이는 모바일에 최적화되지 않은 사이트를 어느 정도 사용할 수 있게 만들었지만, 좁은 너비를 기반으로 하는 미디어 쿼리를 사용하는 반응형 디자인을 망가뜨렸습니다. (14)

width=device-width:

뷰포트 너비를 장치 화면의 물리적 너비(CSS 픽셀 단위)로 설정합니다.(14) 이는 CSS 레이아웃 뷰포트를 장치의 논리적 너비와 효과적으로 일치시킵니다. W3C CSS Viewport Module에 따르면 이는 “실제 뷰포트(actual viewport)“에 영향을 미칩니다.(26)

initial-scale=1:

페이지가 처음 로드될 때 확대/축소 수준을 제어하여 이 초기 확대/축소 수준에서 CSS 픽셀과 장치 독립적 픽셀(또는 OS의 논리적 픽셀) 간의 1:1 관계를 설정합니다.(14) initial-scale=1이 없으면 width=device-width가 설정되어 있어도 브라우저가 여전히 일부 기본 배율을 적용하여 CSS 픽셀이 장치의 논리적 픽셀과 1:1로 정렬되지 않을 수 있습니다.

결합 효과:

두 가지 모두 설정되면(1) CSS 픽셀은 장치 독립적 픽셀(Android의 dp 또는 iOS의 point와 유사)에 해당하도록 의도됩니다. 이 정렬은 window.devicePixelRatio가 OS 수준 배율을 정확하게 반영하는 데 핵심입니다. 그런 다음 devicePixelRatio는 해당 CSS 픽셀(이제 dp/point와 정렬됨)에 사용되는 물리적 픽셀 수를 올바르게 나타냅니다.

확대/축소의 영향:

페이지 확대/축소 (브라우저 확대/축소): devicePixelRatio에 영향을 미칩니다. 확대하면 CSS 픽셀의 크기가 커지므로 devicePixelRatio도 증가합니다.(1)
핀치 줌: 렌더링된 페이지를 확대하지만 CSS 픽셀의 물리적 픽셀에 대한 정의를 변경하지 않으므로 devicePixelRatio에 영향을 미치지 않습니다.(1)
쿼리와의 관련성: 기본 배율(디스플레이에 대해 정적임)과의 비교를 위해 브라우저 확대/축소 없이 initial-scale=1에서 window.devicePixelRatio를 고려합니다.
(30 - WebKit 버그 토론)에서는 window.devicePixelRatio가 구성된 뷰포트(메타 태그)나 페이지 배율에 의존하지 않고 장치의 상수 속성이어야 한다고 언급하며, 이는 기본 비교를 위한 이상적인 상태를 반영합니다.

정보 흐름 (렌더링 엔진에서 JavaScript로):

브라우저(WebView)의 렌더링 엔진(Blink, WebKit)은 기본 운영 체제 또는 임베딩 환경의 정보를 기반으로 장치 픽셀 비율을 결정합니다.
이 값은 window.devicePixelRatio 속성을 통해 JavaScript에 노출됩니다.

DisplayMetrics.density, UIScreen.main.scale, window.devicePixelRatio의 값이 실제로 일치하려면 웹 콘텐츠의 뷰포트가 올바르게 구성되어야 합니다.

DisplayMetrics.density는 dp 배율을 정의하고, UIScreen.main.scale은 point 배율을 정의합니다. 이들은 OS 수준의 논리적 단위입니다.
반면 window.devicePixelRatio는 물리적 픽셀을 CSS 픽셀과 연관시킵니다. 비율이 동등하려면 레이아웃 너비에 사용되는 CSS 픽셀이 OS의 논리적 픽셀(dp 또는 point)에 해당해야 합니다. <meta name="viewport" content="width=device-width, initial-scale=1"> 설정이 이를 달성합니다.(14)
width=device-width는 CSS 픽셀 단위의 레이아웃 너비를 장치의 논리적 너비로 설정하고, initial-scale=1은 이 너비에서 하나의 CSS 픽셀이 하나의 논리적 OS 픽셀과 동일한 유효 크기를 갖도록 보장합니다.

이러한 메타 태그 설정이 없으면 브라우저는 다른 가상 뷰포트 너비(예: 320dp 화면에서도 기본 레이아웃 너비로 980 CSS 픽셀 사용)를 사용할 수 있으므로, 예를 들어 devicePixelRatio가 2이더라도 이는 dp/point에 해당하는 것이 아니라 축소된 CSS 픽셀당 2개의 물리적 픽셀을 의미하게 됩니다.

따라서 meta viewport 태그는 반응형 디자인의 모범 사례일 뿐만 아니라, window.devicePixelRatio가 DisplayMetrics.density 및 UIScreen.main.scale과 비교 가능한 방식으로 장치의 기본 배율을 정확하게 반영하기 위한 근본적인 전제 조건입니다.


6️⃣ 연결고리

각 (OS 정보 기반) 네이티브 ➡️ WebView devicePixelRatio 정보 전달 흐름

Android WebView (Blink 엔진):

Android의 WebView 클래스는 Blink 렌더링 엔진(Chrome과 동일한 엔진)을 사용합니다. 네이티브 애플리케이션 내에서 웹 콘텐츠를 원활하게 렌더링하도록 설계되었습니다.

DisplayMetrics.density를 Blink로 전달:

Android 프레임워크의 일부인 Android WebView 구성 요소는 density 및 densityDpi를 포함한 장치의 DisplayMetrics에 접근할 수 있습니다.

이 정보는 기본 Blink 렌더링 엔진에 전달됩니다. (24)는 “Android의 WebView 클래스와 대부분의 웹 브라우저는 CSS 픽셀 값을 밀도 독립적 픽셀 값으로 변환하므로 웹 페이지가 중간 밀도 화면(약 160dpi)과 동일한 인지 가능한 크기로 나타납니다.”라고 명시합니다. 이 변환은 장치 밀도에 대한 지식을 의미합니다.

(24)은 이를 더 자세히 설명합니다: “WebView는 window.devicePixelRatio를 지원합니다… 이 값이 1.0이면 장치는 중간 밀도 장치로 간주됩니다… 값이 1.5이면 장치는 고밀도 장치로 간주됩니다… Android 브라우저와 WebView가 적용하는 배율은 웹 페이지의 대상 밀도를 기반으로 하며… 기본 대상은 중간 밀도입니다.” 이는 WebView의 window.devicePixelRatio를 DisplayMetrics.density 값(1.0, 1.5 등)으로 정의되는 Android의 밀도 범주(mdpi, hdpi)와 직접 연결합니다.

(33 - Android용 Blink 커밋 로그)는 “장치 픽셀을 직접 사용하여 픽셀 반올림 오류 방지… ViewAndroid는 크기 정보를 장치 픽셀 단위로 받습니다.”라고 언급합니다. 이는 Android의 Blink가 장치 픽셀 정보를 인식하고 이를 사용하여 작동하며, 이는 devicePixelRatio를 올바르게 계산하는 데 필수적임을 나타냅니다.

Blink의 내부 프로세스:

Blink는 네이티브 배율(DisplayMetrics.density에서 파생)을 사용하여 window.devicePixelRatio 값을 설정합니다. 또한 이 요소를 사용하여 CSS 픽셀 하나를 렌더링할 때 사용할 물리적 픽셀 수를 결정합니다(뷰포트에서 initial-scale=1 가정).

CSS 픽셀을 Android의 dp와 정렬 보장:

<meta name="viewport" content="width=device-width, initial-scale=1">을 사용하면 CSS 픽셀이 웹에서 dp에 해당하는 단위가 됩니다. 그러면 window.devicePixelRatio는 물리적 픽셀 / dp에 해당하는 CSS 픽셀이 되며, 이는 DisplayMetrics.density와 수치적으로 동일합니다. 31은 이에 대해 논의합니다: “WebView가 예를 들어 너비가 1920 물리적 픽셀인 경우 CSS 픽셀 단위의 너비는 hdpi 디스플레이에서 1280px여야 합니다.” (1920/1.5=1280). 이는 논리적 CSS 너비 = 물리적 픽셀 너비 / devicePixelRatio 관계를 확인시켜 줍니다.

IOS - WKWebView (WebKit 엔진):

iOS의 최신 WebView 구성 요소로, WebKit 엔진(Safari와 동일)을 사용합니다.

UIScreen.main.scale을 WebKit으로 전달:

WKWebView는 UIView의 하위 클래스입니다. UIView 인스턴스에는 contentScaleFactor 속성이 있으며, 기본적으로 자신이 속한 UIScreen의 scale과 일치합니다.(23)
이 contentScaleFactor(사실상 UIScreen.main.scale)는 WebKit 렌더링 엔진에서 사용할 수 있습니다. (30 - devicePixelRatio에 대한 WebKit 버그 토론): “뷰포트 태그를 기반으로 변경하는 대신 항상 기본 devicePixelRatio를 반환합니다… 이는 iOS 및 Android 브라우저 동작과 일치하며… 제 이해로는 window.devicePixelRatio는 구성된 뷰포트에 의존하지 않고 장치의 상수 속성이어야 합니다.” 이는 iOS에서 window.devicePixelRatio가 UIScreen.main.scale로 설정됨을 강력하게 시사합니다.

WebKit의 내부 프로세스:

WebKit은 UIScreen.main.scale 값(예: 1.0, 2.0, 3.0)을 가져와 window.devicePixelRatio를 이 값으로 설정합니다.
이 배율을 사용하여 CSS 픽셀을 렌더링합니다. 예를 들어, @2x 장치에서 1 CSS 픽셀(initial-scale=1 사용)은 2x2 물리적 픽셀 그리드를 사용하여 렌더링됩니다.

WKWebViewConfiguration의 역할:

WKWebViewConfiguration은 데이터 저장소, 사용자 스크립트 주입, 미디어 재생 기본 설정, 프로세스 풀 관리 등 WKWebView의 다양한 속성을 생성 시 설정하는 데 사용되는 객체입니다.(29)
ignoresViewportScaleLimits와 같은 속성이 있지만(29), 이는 사용자가 시작한 페이지 배율/확대/축소와 관련이 있으며 하드웨어에서 파생된 기본 장치 픽셀 비율과는 관련이 없습니다.
핵심 devicePixelRatio는 시스템의 화면 배율(UIScreen.main.scale)에 의해 결정되며, 이는 WKWebView가 작동하는 디스플레이 환경의 고유한 속성이며, 이 특정 목적을 위해 WKWebViewConfiguration을 통해 구성 가능한 항목이 아닙니다.

CSS 픽셀을 iOS의 points와 정렬 보장:

<meta name="viewport" content="width=device-width, initial-scale=1">을 사용하면 하나의 CSS 픽셀이 웹에서 iOS point에 해당하는 단위가 됩니다.
그러면 window.devicePixelRatio는 물리적 픽셀 / point에 해당하는 CSS 픽셀이 되며, 이는 UIScreen.main.scale과 수치적으로 동일합니다.
Android의 WebView(Blink를 통해)와 iOS의 WKWebView(WebKit을 통해) 모두 자체 배율을 만드는 것이 아니라 호스트 운영 체제에서 제공하는 배율을 반영합니다.
WebView는 네이티브 애플리케이션의 일부로 웹 콘텐츠를 렌더링하도록 설계된 임베디드 구성 요소입니다. 원활한 사용자 경험을 위해서는 웹 콘텐츠가 인지되는 크기와 선명도 측면에서 네이티브 UI 요소와 시각적으로 정렬되어야 합니다.
WebView가 다른 배율 논리를 사용하면 웹 콘텐츠가 인접한 네이티브 구성 요소에 비해 불균형하게 크거나 흐릿하게 보일 수 있습니다.
따라서 렌더링 엔진(Blink/WebKit)은 임베딩 환경에서 네이티브 배율(Android의 DisplayMetrics.density, iOS의 UIScreen.scale)을 쿼리하거나 수신하도록 설계되었습니다.(23) 이러한 위임은 WebView 내부의 window.devicePixelRatio가 네이티브 OS 배율을 미러링하도록 보장하는 직접적인 메커니즘이며, 사용자가 문의하는 동등성으로 이어집니다.
렌더링 엔진은 OS의 밀도 정보를 웹 좌표계로 충실히 변환하는 다리 역할을 합니다.


7️⃣ 정리

DisplayMetric.density, UIScreen.main.scale, window.devicePixelRatio는 동일한 비율!

공통 목표: 논리적 단위의 일관된 물리적 크기

세 플랫폼 모두 기본 논리적 단위(dp, point, CSS 픽셀)가 픽셀 밀도에 관계없이 모든 화면에서 거의 동일한 물리적 영역을 차지하도록 노력합니다.(7) 이를 통해 100 논리적 단위 너비로 정의된 버튼이 사용자에게 일관된 크기로 표시됩니다.

DisplayMetrics.density (Android):

장치 DPI / 160 DPI.
이는 장치의 물리적 픽셀 밀도 대 기준 논리적 밀도의 비율입니다.
1 dp를 1 논리적 픽셀 단위로 간주하면 density는 1 dp 단위에 맞는 (160 DPI에 해당하는) 물리적 픽셀 수(배율 조정됨)입니다.
또는 더 직접적으로, dp 단위를 물리적 픽셀로 변환하기 위해 배율 조정되는 요소입니다: 물리적 픽셀 = dp 값 × density

UIScreen.main.scale (iOS):

물리적 픽셀 / 포인트.
이는 주어진 차원에 대한 물리적 픽셀 대 논리적 포인트의 직접적인 비율입니다.

window.devicePixelRatio (Webview):

물리적 픽셀 / CSS 픽셀
(뷰포트 initial-scale=1 및 width=device-width가 설정되어있다는 전제 하에 CSS 픽셀이 dp/point와 유사한 논리적 픽셀처럼 동작할 때)

WebView는 호스트 OS 배율 존중:

6️⃣ 에서 설명했듯이 WebView(Blink를 사용하는 Android WebView, WebKit을 사용하는 iOS WKWebView)는 호스트 운영 체제에서 네이티브 배율을 가져오도록 설계되었습니다.
Android WebView는 DisplayMetrics.density를 사용하여 Blink에 정보를 제공하고, Blink는 window.devicePixelRatio를 이 값과 일치하도록 설정합니다.(24)
iOS WKWebView는 UIScreen.main.scale(UIView.contentScaleFactor를 통해)을 사용하여 WebKit에 정보를 제공하고, WebKit은 window.devicePixelRatio를 이 값과 일치하도록 설정합니다.(23)

동등성:

dp(Android), point(iOS), 올바르게 구성된 CSS 픽셀(Web)은 모두 장치 독립적인 논리적 단위로 설계되었기 때문에, 그리고 배율(density, scale, devicePixelRatio)은 모두 물리적 픽셀 / 논리적 단위의 비율로 정의되기 때문에, 그리고 WebView는 window. devicePixelRatio에 대해 호스트 OS의 배율을 채택하기 때문에, 따라서 DisplayMetrics.density(Android), UIScreen.main.scale(iOS), window.devicePixelRatio(해당 OS의 WebView에서 적절한 뷰포트 설정 사용)는 동일한 숫자 값을 갖게 됩니다. 이들은 모두 각 환경에서 하나의 논리적 픽셀을 렌더링하는 데 사용되는 물리적 장치 픽셀 수를 나타냅니다.

8️⃣ 결론

DisplayMetrics.density, UIScreen.main.scale, window.devicePixelRatio는 모두 논리적 디자인 단위와 물리적 화면 픽셀 간의 간극을 메우기 위해 설계된 동일한 필수 배율 메커니즘의 표현입니다.
운영체제는 디스플레이의 물리적 특성을 결정하고 네이티브 배율을 계산합니다. 이 요소는 네이티브 애플리케이션에 노출됩니다.
웹 콘텐츠가 WebView 내에 표시될 때 렌더링 엔진(Blink 또는 WebKit)은 호스트 OS에서 이 네이티브 배율을 쿼리하고 이를 사용하여 window.devicePixelRatio를 설정합니다.
웹 페이지 뷰포트(<meta name="viewport">)의 적절한 구성은 CSS 픽셀이 OS의 논리적 픽셀과 정렬되어 비교를 의미 있게 만드는 데 필수적입니다.
이러한 동등성과 기본 메커니즘을 이해하는 것은 여러 모바일 플랫폼에서 네이티브 및 웹 구성 요소에 걸쳐 일관된 사용자 경험을 만들고자 하는 개발자에게 매우 중요합니다.
이를 통해 장치의 화면 밀도에 관계없이 UI 요소의 예측 가능한 크기 조정과 선명한 렌더링이 가능해집니다.
밀도 배율에 대한 이러한 통합된 접근 방식은 하이브리드 애플리케이션 개발을 단순화하고 웹 콘텐츠가 네이티브 애플리케이션 셸 내에서 조화롭게 통합되도록 보장합니다.
